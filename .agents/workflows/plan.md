# 기획 워크플로우 (/plan)

이 워크플로우는 새로운 기능을 개발하기 전, 요구사항을 명확히 하고 실행 가능한 최소 단위의 작업 계획을 수립하기 위한 가이드라인입니다.
요구사항 인터뷰 → 요구사양 문서 작성 → 기술 검토 → 작업 계획 작성 → 계획 리뷰 → 문서 저장 순으로 진행합니다.

---

## 1단계: 요구사항 인터뷰

사용자와 대화를 통해 구현하려는 기능의 명확한 요구사항을 파악합니다.

- **목적 및 배경**: 왜 이 기능이 필요한지 확인합니다.
- **핵심 요구사항**: 반드시 포함되어야 할 기능들을 목록화합니다.
- **완료 조건 (DoD)**: 어떤 상태가 되었을 때 작업이 완료된 것으로 간주할지 합의합니다.
- **제약 사항**: 기술적/비즈니스적 제약 사항이 있는지 확인합니다.

> **참고**: 인터뷰 내용을 요약하여 사용자에게 확인을 받은 후 2단계로 진행합니다.

---

## 2단계: 요구사양 문서(`requirements.md`) 작성 및 기술 검토

인터뷰 결과를 바탕으로 요구사양 문서를 작성하고, 서브에이전트에게 기술적 검토를 받습니다.

### 2.1 요구사양 문서 작성
`requirements.md` 파일을 작성합니다. 이 문서에는 다음 내용이 포함되어야 합니다:
- 기능 개요 및 목적
- 상세 기능 요구사항 목록
- 완료 조건 (Definition of Done)
- 관련 기존 코드 및 영향 범위 분석 기초 데이터

### 2.2 테크니컬 컨설턴트 서브에이전트 실행
작성된 `requirements.md`를 바탕으로 **테크니컬 컨설턴트 서브에이전트**를 실행합니다. 상세 지침은 [.agents/roles/technical_consultant.md](.agents/roles/technical_consultant.md)를 참조하십시오.
- **검토 항목**: 아키텍처 적합성, 기술 스택 연계성, 잠재적 리스크.
- **결과 활용**: 지적된 리스크나 제안을 `requirements.md`에 반영하여 보완합니다.

---

## 3단계: 작업 계획 문서(`task_plan.md`) 작성

기술 검토가 완료된 요구사항을 바탕으로 실제 실행 가능한 `task_plan.md`를 작성합니다.

### 작성 원칙
- **서브에이전트 단위**: 1개의 작업(Task)은 1개의 서브에이전트가 한 번의 실행으로 완결할 수 있는 적절한 크기로 쪼갭니다.
- **TDD 준수**: 모든 작업은 반드시 TDD 절차를 포함해야 합니다.
  - `[RED]`: 실패하는 테스트 케이스 정의
  - `[GREEN]`: 테스트를 통과하는 최소 기능 구현
  - `[REFACTOR]`: 코드 품질 개선
- **독립성**: 각 작업은 가급적 독립적으로 검증 가능해야 합니다.

---

## 4단계: 작업 계획 리뷰

작성된 `task_plan.md`가 적절하게 설계되었는지 **플랜 리뷰어 서브에이전트**를 통해 검토받습니다. 상세 지침은 [.agents/roles/plan_reviewer.md](.agents/roles/plan_reviewer.md)를 참조하십시오.

### 리뷰 포인트
- **TDD 준수 여부**: 모든 작업에 RED/GREEN/REFACTOR 단계가 명확히 명시되어 있는가?
- **작업 단위의 적절성**: 각 작업이 서브에이전트 1개가 처리하기에 너무 크거나 작지 않은가? (Granularity)
- **실행 가능성**: 작업 간의 의존성이 명확하고 순서가 올바른가?

### 검토 결과 처리
- 리뷰어의 피드백을 반영하여 `task_plan.md`를 수정합니다.
- 서브에이전트로부터 '승인' 또는 '적절' 판정을 받을 때까지 보완합니다.

---

## 5단계: 문서 최종 저장 및 확정

최종 확정된 문서들을 저장하여 향후 작업 실행 워크플로우에서 참조할 수 있도록 합니다.

- **저장 위치**: 프로젝트 루트 또는 지정된 기획 문서 디렉토리 (예: `docs/planning/`)
- **대상 파일**:
  1. `requirements.md`: 요구사항 및 기술 검토 결과 포함
  2. `task_plan.md`: 리뷰가 완료된 TDD 기반 작업 계획서

이 문서들은 추후 생성될 '작업 실행 워크플로우'에서 서브에이전트에게 전달할 핵심 컨텍스트로 사용됩니다.
