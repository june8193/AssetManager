# 개발 워크플로우 (/develop)

이 워크플로우는 새로운 기능 개발 또는 변경 작업을 시작할 때 준수해야 하는 단계별 가이드라인입니다.
요구사항 인터뷰 → 기술 컨설팅 → 계획 작성 → 플랜 리뷰 → 플랜 확정 → 코드 작업 순으로 진행합니다.

---

## 1단계: 요구사항 인터뷰

사용자와 대화를 통해 요구사항을 명확히 파악합니다.

- 구현하려는 기능의 **목적과 배경**을 확인합니다.
- **핵심 기능 목록**을 사용자와 함께 정의합니다.
- **완료 조건(Definition of Done)** 을 명확히 합의합니다.
- 아래 항목에 대해 질문하여 모호함을 제거합니다:
  - 사용자 시나리오 (누가, 어떤 상황에서, 무엇을 하는가)
  - 비기능 요구사항 (성능, 보안, 확장성 등)
  - 기존 코드/데이터와의 연관성 및 영향 범위
- 인터뷰 결과를 요약하여 사용자에게 확인을 받습니다.

> **참고**: 인터뷰가 끝나기 전까지 다음 단계로 진행하지 않습니다.

---

## 2단계: 테크니컬 컨설턴트 서브에이전트 실행

요구사항을 바탕으로 **테크니컬 컨설턴트 서브에이전트**를 실행하여 기술적 설계 방향을 검토받습니다.

### 서브에이전트 역할
테크니컬 컨설턴트는 아래 관점에서 기술적 검토를 수행합니다:
- **아키텍처 적합성**: 요구사항에 맞는 설계 패턴 및 아키텍처 방향 제안
- **기술 스택 검토**: 현재 프로젝트 기술 스택과의 연계성 및 적합한 라이브러리/도구 추천
- **리스크 분석**: 구현 시 예상되는 기술적 난이도, 잠재적 문제점 및 병목 구간 식별
- **대안 검토**: 요구사항을 충족하는 복수의 기술적 접근 방법과 장단점 비교

### 서브에이전트 프롬프트 구성 방법
서브에이전트를 실행할 때 다음 정보를 포함한 상세한 프롬프트를 작성합니다:

```
당신은 시니어 테크니컬 컨설턴트입니다. 아래 요구사항과 프로젝트 컨텍스트를 분석하여
기술적 설계 방향과 권장 아키텍처를 제시해 주세요.

## 프로젝트 컨텍스트
[현재 프로젝트의 기술 스택, 아키텍처 개요, 주요 모듈 구조를 기술]

## 요구사항 (인터뷰 결과 요약)
[1단계에서 확인된 기능 목록, 완료 조건, 비기능 요구사항 등을 기술]

## 검토 요청 사항
1. 권장 아키텍처 및 설계 패턴
2. 구현에 필요한 기술/라이브러리 목록과 선택 이유
3. 예상되는 기술적 리스크 및 해결 방안
4. 고려해야 할 대안적 접근 방법과 장단점

## 출력 형식
- 한국어로 작성합니다.
- 각 항목별로 명확한 근거와 함께 구체적인 권고안을 제시합니다.
- 최종적으로 '권장 방향 요약'을 3~5줄로 정리합니다.
```

### 결과 활용
- 서브에이전트의 검토 결과를 사용자에게 공유합니다.
- 필요 시 사용자와 결과를 논의하고, 기술 방향을 확정합니다.

---

## 3단계: 구현 계획 작성

1~2단계의 결과를 바탕으로 **`implementation_plan.md`** 를 작성합니다.

### 계획서 포함 항목
계획서는 아래 구조를 반드시 포함해야 합니다:

#### 개요
- 구현 목표 및 배경
- 완료 조건 (Definition of Done)

#### 기술 설계
- 채택할 아키텍처 및 설계 방향 (2단계 컨설팅 결과 반영)
- 변경/추가될 파일 목록 (`[NEW]`, `[MODIFY]`, `[DELETE]` 구분)

#### 태스크 목록
각 태스크는 아래 원칙에 따라 분리합니다:
- **독립적으로 검증 가능한 단위**로 분리 (하나의 태스크 = 하나의 명확한 검증 기준)
- **TDD 절차 명시**: 각 태스크에 대해 Red → Green → Refactor 단계를 구분하여 기술
  - `[RED]` 먼저 작성할 실패 테스트 케이스 목록
  - `[GREEN]` 테스트를 통과시키기 위한 최소 구현 내용
  - `[REFACTOR]` 리팩토링 포인트
- 태스크 간 **의존성(선행 태스크)** 을 명시

#### 검증 계획
- 단위 테스트 실행 방법 (`pytest` 커맨드 등)
- E2E 테스트 시나리오 목록

---

## 4단계: 플랜 리뷰어 서브에이전트 실행

작성된 계획서를 **플랜 리뷰어 서브에이전트**가 검토합니다.

### 서브에이전트 역할
플랜 리뷰어는 아래 관점에서 계획서를 검토합니다:

1. **태스크 분리 적절성**
   - 각 태스크가 독립적으로 완료/검증 가능한 단위인가?
   - 태스크가 너무 크거나(분리 필요) 너무 작아(통합 가능) 불필요하게 복잡하지 않은가?
   - 태스크 간 의존성이 명확히 정의되어 있는가?

2. **TDD 계획 검증**
   - 각 태스크에 `[RED]` 테스트 케이스가 구체적으로 기술되어 있는가?
   - 테스트가 실제 비즈니스 로직을 검증하는가? (단순 구현 확인이 아닌 동작 검증)
   - 데이터베이스 격리 등 테스트 환경이 고려되어 있는가?
   - E2E 테스트 시나리오가 실제 사용자 시나리오를 커버하는가?

3. **완료 조건 명확성**
   - 각 태스크의 완료 기준이 객관적으로 측정/확인 가능한가?

### 서브에이전트 프롬프트 구성 방법

```
당신은 시니어 개발 플랜 리뷰어입니다. 아래 구현 계획서를 검토하고, 태스크 분리의 
적절성과 TDD 계획의 충실도를 평가해 주세요.

## 프로젝트 규칙 (GEMINI.md 요약)
- TDD 방식으로 개발: Red(실패 테스트 작성) → Green(최소 구현) → Refactor
- 테스트 격리: 테스트 시 운영 DB에 영향을 주지 않아야 함 (인메모리 DB 사용)
- 백엔드 테스트: pytest, tests/ 디렉토리, test_ 접두사
- 프론트엔드 테스트: Vitest + React Testing Library, .test.jsx 접미사

## 검토할 구현 계획서
[3단계에서 작성한 implementation_plan.md 전체 내용 삽입]

## 검토 항목
1. 태스크 분리 적절성 평가 (각 태스크별로 피드백)
2. TDD 계획 충실도 평가 (RED 테스트 케이스의 구체성, 검증 가능성)
3. 누락된 엣지 케이스나 테스트 시나리오
4. 전체 계획의 실행 가능성 평가

## 출력 형식
- 한국어로 작성합니다.
- 각 태스크에 대해 '✅ 적절', '⚠️ 보완 필요', '❌ 재작성 필요' 중 하나로 판정합니다.
- 보완/재작성이 필요한 경우 구체적인 개선 방향을 제시합니다.
- 마지막에 '종합 의견'을 작성하고 계획서 승인 여부(승인 / 조건부 승인 / 반려)를 명시합니다.
```

### 결과 처리
- **승인**: 5단계(플랜 확정)으로 진행합니다.
- **조건부 승인**: 제시된 보완 사항을 계획서에 반영한 후 5단계로 진행합니다.
- **반려**: 계획서를 리뷰 결과에 따라 수정한 후 4단계를 재실행합니다.

---

## 5단계: 플랜 확정 및 사용자 승인

수정된 최종 계획서를 사용자에게 제시하고 승인을 요청합니다.

- `implementation_plan.md` 아티팩트를 업데이트하고 사용자에게 공유합니다.
- 플랜 리뷰어의 검토 결과 요약을 함께 제공합니다.
- 사용자의 **명시적 승인**을 받은 후에만 다음 단계로 진행합니다.

> **주의**: 사용자 승인 없이 코드 작업을 시작하지 않습니다.

---

## 6단계: 코드 작업 진행

사용자 승인이 완료된 후 코드 작업을 시작합니다.

- `task.md` 아티팩트를 생성하고 태스크 목록을 관리합니다.
- 계획서의 태스크 순서와 의존성을 준수하며 진행합니다.
- 각 태스크는 **TDD 절차(Red → Green → Refactor)** 를 엄격히 따릅니다.
- 태스크 완료 시마다 `task.md`를 업데이트합니다.
- 예상치 못한 기술적 이슈로 계획 변경이 필요한 경우, `implementation_plan.md`를 수정하고 사용자에게 보고한 후 계속 진행합니다.
- 모든 태스크 완료 후 `walkthrough.md`를 작성하여 변경 사항을 요약합니다.
