# 개발 작업 계획 수립 스킬 (Task Planner)

이 스킬은 복잡한 개발 작업을 체계적으로 계획하고, 서브에이전트가 수행 가능한 단위로 작업을 분할하여 문서화하는 과정을 안내합니다.

## 개발 절차

이 스킬이 활성화되면 반드시 다음의 순서를 엄격히 준수하여 진행하십시오.

### 1. 요구사항 파악 (Requirement Gathering)
- 사용자에게 구현하고자 하는 기능이나 해결하려는 문제에 대한 요구사항을 상세히 물어봅니다.

### 2. 코드베이스 탐색 (Research)
- 관련 있는 코드베이스를 탐색하여 현재 구조와 영향 범위를 분석합니다.
- `grep_search`, `glob`, `read_file` 등을 사용하여 기술적 맥락을 파악합니다.

### 3. 추가 인터뷰 (Interview)
- 코드 분석 결과를 바탕으로 요구사양과 관련하여 궁금한 점이나 명확히 해야 할 부분을 사용자에게 다시 인터뷰하여 정보를 수집합니다.

### 4. 기술적 요구사항 문서 작성 및 검토 (RQMT Doc)
- 수집된 정보를 바탕으로 기술적 요구사양 문서(`rqmt_*.md`)를 작성합니다.
- **문서 위치**: `docs/planning/rqmt_[기능명].md`
- 작성 후 `@technical_consultant` 서브에이전트에게 검토를 요청합니다.
- 검토 의견을 반영하여 문서를 보완합니다.

### 5. 사용자 승인 (User Approval - RQMT)
- 완성된 요구사양 문서에 대해 사용자의 승인을 받습니다.

### 6. 작업 계획 문서 작성 및 검토 (Plan Doc)
- 승인된 요구사양을 바탕으로 구체적인 작업 계획 문서(`plan_*.md`)를 작성합니다.
- **문서 위치**: `docs/planning/plan_[기능명].md`
- **작업 분할 및 완결성 원칙**:
  - 개별 작업은 1개의 서브에이전트가 완결성 있게 수행할 수 있는 단위로 쪼개야 합니다.
  - **TDD 통합**: `[RED]`, `[GREEN]`, `[REFACTOR]` 단계를 개별 태스크로 분리하지 마십시오. 하나의 태스크 내에서 기능 구현, 테스트, 검증, 커밋(또는 커밋 준비)까지 완결되어야 합니다.
  - 각 태스크는 독립적으로 검증 가능해야 하며, 수행 후 시스템이 안정적인 상태(테스트 통과)를 유지해야 합니다.
- 작성 후 `@plan_reviewer` 서브에이전트에게 검토를 요청합니다.
- 검토 의견을 반영하여 계획을 보완합니다.

### 7. 최종 승인 및 저장 (Final Approval & Save)
- 작업 계획 문서에 대해 사용자의 최종 승인을 받습니다.
- 모든 문서가 `work` 폴더에 올바른 prefix(`rqmt_`, `plan_`)로 저장되었는지 확인합니다.

## 문서 명명 규칙
- 요구사양 문서: `docs/planning/rqmt_[주요기능명].md`
- 작업계획 문서: `docs/planning/plan_[주요기능명].md`

## 서브에이전트 활용 가이드
- **기술적 요구사항 검토**: `technical_consultant`
- **작업 계획 검토**: `plan_reviewer`
- **개별 작업 수행**: 작업 계획 문서의 각 항목은 서브에이전트에게 위임할 수 있도록 명확한 목표와 범위를 가져야 합니다.
